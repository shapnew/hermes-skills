# Brainstorming — 需求打磨与设计确认

## 核心规则（HARD-GATE）

> **在呈现设计并获得用户批准之前，严禁实施任何动作。**
> 包括但不限于：写代码、改配置、创建文件、修改数据库、安装软件。
> 这条规则对 **每个任务都生效**，不管你觉得多简单。

## 何时触发

以下任何场景出现时，必须激活 brainstorming：

- 用户提出一个新功能、新需求、新项目
- 用户说"帮我做个XX"、"我想加个XX"
- 用户描述了一个模糊的想法需要明确
- 一个现有任务/问题有多个可行方案可选
- 任何需要你做决策性工作之前的场景

## 检查清单（按顺序执行）

### 1️⃣ 探索上下文
- 检查现有项目文件、配置、文档
- 如果是已有项目，看最近的提交记录、存在的技能
- 了解当前状态：已有啥？缺啥？

### 2️⃣ 理解真实需求
- 一次只问 **一个问题**
- 优先用选择题让用户选
- 关注：**目的**（为什么做）、**约束**（时间/预算/技术限制）、**成功标准**（怎么做才算好）

### 3️⃣ 探索方案（如果问题有足够复杂度）
- 提出 2-3 种不同方案，附带优缺点
- 先说你的推荐方案 + 理由
- 让用户选择方向

### 4️⃣ 呈现设计
- 分节呈现，每节控制在一个概念
- 每节完成后问"这样对吗？"
- 覆盖：架构、组件、数据流、边界情况

### 5️⃣ 确认后写设计文档
- 写入 `.hermes/plans/` 目录下的设计文档
- 命名格式：`YYYY-MM-DD-<主题>-design.md`
- 内容包括：背景、需求、方案对比、选定方案、实施要点

### 6️⃣ 规格自检
- 有占位符/待办未完成吗？
- 各节之间矛盾吗？
- 范围聚焦吗？
- 有没有模糊的表述？

### 7️⃣ 用户最后确认
- "设计文档已写好，路径：xxx，请过目。确认没问题的话我开始做。"

## 设计原则

- **一次只问一个问题** — 不要同时扔多个问题
- **优先选择题** — "A 还是 B？"比"你说咋整？"高效
- **六亲不认的 YAGNI** — 问"这个真的需要吗？"，砍掉任何不必要的功能
- **渐进确认** — 设计完一段确认一段，不要攒到最后
- **简单优先** — 如果两种方案差不多，选更简单的

## 反模式：要警惕的念头

| 你可能会想 | 实际该做的 |
|---|---|
| "这很简单，直接干吧" | 再简单的也要先确认设计 |
| "我边做边想" | 停下来，先想清楚 |
| "先改这一小点" | 任何改动前都要确认 |
| "我知道他要什么" | 你不知道，问 |
| "以前做过类似的" | 上下文不同，重走流程 |

## 与 Hermes 现有工具的配合

- **todo 工具**：复杂任务先建 task list，标 pending
- **delegate_task**：确认设计后，通过子 agent 并行实现
- **terminal**：只在确认后可执行
- **write_file / patch**：只在确认后修改文件

## 特别说明

这个 skill 适用于 **小灵（Hermes）** 的生态。它在技能系统加载时自动触发。
当多个技能可能相关时，brainstorming 有最高优先级——先确定做什么，再决定怎么做。

触发短语参考（用户可能用这些话，一旦听到就要进 brainstorming）：
- "帮我做个……"
- "我想加个……"
- "有没有什么方法可以……"
- "你看这样行不行……"
- "我想搞个……"
- 用户给了模糊的想法
- 涉及系统变更（安装/修改配置/文件操作）
